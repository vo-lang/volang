# BlockKart Whiteboard / Volang Runtime Investigation

Date: 2026-04-29
Status: resolved in the accompanying Volang changes

This note records the root-cause chain behind the BlockKart quickplay whiteboard failures and the follow-up GC/JIT runtime fixes. It is an issue-specific handoff note; canonical bytecode/runtime behavior belongs in specs and source comments.

## Symptoms

BlockKart quickplay could compile and start, then fail in the browser runner with one of these runtime failures:

```text
primitive3d.ShapeRegistry.Desc: unknown shape
mark_gray: invalid GcRef ... non-GcRef value in GcRef-typed slot
RuntimeError: memory access out of bounds
```

The first visible app-level symptom was often a blank/white runner surface.

## Primary Root Cause: `PtrNew` Slot Count Truncation

The main BlockKart corruption came from bytecode encoding, not from `primitive3d` shape registration.

`PtrNew` previously stored the heap object slot count in `Instruction.flags`, which is `u8`. Large value objects with more than 255 slots were truncated when codegen emitted heap allocation. BlockKart's larger primitive/HUD/world objects could therefore allocate too few heap slots, and later writes corrupted adjacent GC/runtime state. One observed downstream effect was `primitive3d.ShapeRegistry.Desc: unknown shape`.

The fix changes the canonical instruction format:

```text
PtrNew: a=dst, b=meta register, c=slot count, flags=reserved
```

The VOB serialization version was bumped from 1 to 2. No compatibility decoding was added; old bytecode should be regenerated with the current compiler.

Regression coverage:

```text
lang/test_data/bugs/2026_04_30_ptrnew_large_struct_slots.vo
```

That test compiles a struct with 261 heap slots and checks the formatted bytecode reports `slots=261`, not a truncated `u8` value.

## Secondary GC Finding

During the investigation, a temporary "new objects are born black during marking" approach caused a separate GC regression:

```text
mark_gray: invalid GcRef ... non-GcRef value in GcRef-typed slot
```

That approach was incorrect because Vo object initialization does not guarantee every reference write goes through a barrier before the object is fully initialized.

The final GC behavior is:

- new objects still use the current white color;
- objects allocated during `Propagate` or `Atomic` are queued gray, so initialized reference slots are scanned before sweep;
- roots are rescanned at the atomic boundary because mutable stack/global roots can become live after `start_cycle`;
- the sweep write barrier rescues old-white children written during sweep.

Regression coverage:

```text
lang/test_data/bugs/2026_04_29_gc_late_global_root.vo
lang/test_data/gc_incremental_sweep_barrier.vo
vo-runtime gc::tests::
```

## Stack Overflow Boundary

The `RuntimeError: memory access out of bounds` class was a separate VM/JIT boundary issue. Legal infinite recursion should become a recoverable Vo stack-overflow panic, not a host panic or wasm memory trap.

The fix adds explicit fiber stack/frame capacity errors and routes VM call setup through fallible APIs. JIT direct calls now have both:

- a slot budget for large-frame direct native chains;
- a call-depth budget for tiny or zero-local recursive chains.

JIT entry and side-exit frame materialization now convert capacity failures into the same recoverable runtime panic path.


## Studio/Quickplay Notes

Studio VFS compile-cache fingerprints now include a schema version, compiler package version, and Studio WASM build id. This prevents stale compiled VOBs from surviving bytecode/runtime format changes.

BlockKart quickplay packaging is generated by:

```sh
cd studio
npm run package:blockkart
```

The script embeds the BlockKart project source at the current BlockKart `HEAD` and packages dependency source/artifacts from the locked module cache.
