# Volang GC v1 Plan

## Goal

Build the current incremental mark-sweep collector into a correct, observable, low-pause GC suitable for interactive runtimes and ordinary 60 FPS games.

This is not a generational GC project and not an allocator rewrite. The goal is to finish the current collector as a coherent system: explicit invariants, measurable pauses, complete root-dirty integration, and focused stress coverage.

## Non-goals

- Do not specialize GC behavior for BlockKart or any one app.
- Do not skip root scans unless a caller provides an explicit freshness proof.
- Do not introduce a nursery or remembered-set based generational collector in this phase.
- Do not rewrite the object allocator before telemetry proves allocation itself is the bottleneck.
- Do not split atomic phase without first measuring its real cost and correctness constraints.

## Invariants

- `Gc::step` is conservative by default. It must remain correct when roots may have changed.
- Optimized root scanning uses `GcRootState::StableSinceLastScan` only when the caller can prove roots have not changed since the collector's last root scan.
- During sweep, a newly reachable old-white object must be rescued before sweep reaches it.
- Heap write barriers handle propagate and sweep correctly; root changes are tracked by root-dirty state, not heap barriers.
- Stress mode must expose missing root/write barriers, not hide them behind frame-specific workarounds.

## Acceptance Criteria

- `cargo test -p vo-runtime` and `cargo test -p vo-vm` pass.
- JIT regression `bugs/2026_04_29_push_borrowed_call_frame_oob.vo` passes in release JIT.
- GC perf harness has root-heavy, sparse-root, interior-root, dead-sweep, and live-graph scenarios.
- Runtime GC telemetry reports phase, root scan calls, object scans, finalizers, sweep frees, work bytes, heap bytes, debt, and queue lengths per step.
- VM GC telemetry reports whether roots were full-scanned, dirty-scanned, or stable-skipped, plus dirty fiber counts.
- `./d.py gc-perf --release --objects=50000` remains sub-millisecond to low-millisecond for root-heavy scenarios.
- `./d.py gc-perf --release --objects=200000` should keep steady-state GC slices below a frame budget under normal machine load, and any outlier must be investigated with a focused rerun before treating it as collector work.

## Implementation Plan

1. Add GC core telemetry.
   - Store the last step's phase transition, work bytes, root scan calls, object scan calls, finalized objects, freed bytes, heap bytes, debt, and gray queue lengths.
   - Keep timing out of `vo-runtime` so the core remains no_std-compatible.

2. Connect telemetry to the perf harness.
   - Print root scan calls and object/finalize counts from collector telemetry instead of ad hoc counters where possible.
   - Add JSON fields needed for trend comparisons.

3. Complete VM dirty-root accounting.
   - Keep `Gc::step` conservative.
   - In the VM scheduler, use dirty-fiber scans during sweep when only the currently run fiber changed.
   - Mark dirty-all for host events, I/O, island commands, endpoint registry changes, sentinel errors, and other non-current-fiber root mutations.
   - Add tests for each dirty source that can make an old-white object reachable during sweep.

4. Add runtime-facing telemetry accessors.
   - Expose last GC step telemetry through `Vm`.
   - Keep API small and copyable so app/runtime hosts can poll it without allocations.

5. Evaluate atomic phase.
   - Use telemetry to decide whether atomic phase is a real pause source.
   - Only then design bounded atomic processing if needed.

## Current Deferred Work

- Generational GC with nursery and remembered sets.
- Size-class allocator or bump allocation.
- Per-stack-slot dirty bitsets.
- Cross-island distributed GC.

## Implementation Status

Completed in the first implementation pass:

- Runtime `GcStepStats` records phase transitions, root scan calls/skips, object scans, finalizers, sweep frees, work bytes, heap bytes, debt, gray queue lengths, and cycle start/finish flags.
- `gc-perf` reports collector telemetry instead of ad hoc callback counters, and includes root-heavy, sparse-root, interior-root, dead-sweep, and live-chain scenarios.
- Runtime `step_with_root_scanner` supports bounded root-scan chunks for hosts that can prove root-set stability or provide their own dirty/restart protocol.
- VM root scans now use precise snapshots and cursors on top of `step_with_root_scanner`, so large root sets are split across bounded root-work chunks instead of requiring one full root scan in a single GC step.
- VM root snapshots carry a dirty epoch. If roots change while any bounded root scan is in progress, including start-cycle, atomic, or sweep scans, the VM discards the stale snapshot and restarts from a fresh root view before the collector can rely on it.
- VM `VmGcStepStats` reports full-root scans, dirty-root scans, stable root skips, dirty-all state, dirty fiber count, and raw GC step stats.
- VM scheduling now uses dirty-fiber sweep scans after a mutating fiber boundary and dirty-all scans for host/I/O/island/endpoint root mutations.
- Stress and regression coverage includes late sweep roots, stale pending start-cycle/atomic root snapshots, stable sweep skip, endpoint/queue root dirtying paths, budgeted VM full-root scans, and the JIT borrowed call-frame OOB regression.

Validated commands:

- `cargo test -p vo-runtime`
- `cargo test -p vo-vm`
- `./d.py gc-perf --release --objects=50000`
- `./d.py gc-perf --release --objects=200000`
- `./d.py test jit --release bugs/2026_04_29_push_borrowed_call_frame_oob.vo`
- `npm run build:wasm` from `studio`

Measured result:

- `50k` object scenarios stay below `1ms` max step time on this machine in the current release run.
- `200k` artificial scenarios no longer force one full root scan in a single collector step when using the bounded root scanner. The current release run stayed at `max_ms <= 1.532` across `dead-sweep`, `live-chain`, `root-table`, `sparse-root-table`, and `interior-root-table`.
- VM root scans are budgeted too. `full_vm_root_scan_is_budgeted` verifies that 2048 stack roots are split into 8KB root-work chunks before object marking resumes.
